# RFC-0001：Sango AI 驱动开放世界愿景

- 状态：Proposed
- 决策人：Project Owner
- 评审人：Codex、Antigravity、ChatGPT Remote Reviewer
- 影响范围：产品愿景、世界模拟、AI 架构、内容生产、Three.js 场景演进
- 本 PR 性质：仅文档，不修改玩法代码、资产或当前 Phase 1 验收范围

## 1. 决策摘要

Sango 将以现有原创三国回合策略底座为核心，逐步演进为一个 **AI 驱动、持续演化、可探索的三国开放世界**。

这里的“开放世界”首先指：

1. 世界在玩家不干预时仍持续运行；
2. 势力、城池、武将、百姓和事件彼此影响；
3. 武将具有目标、关系、记忆、立场和有限认知；
4. 玩家可从战略地图进入局部场景，亲自交涉、治理、侦察和行动；
5. 历史只提供初始条件，之后由系统与玩家共同生成不同历史。

它不等于在第一阶段立即制作一张巨大、无缝、可步行的 3D 中国地图。

## 2. 产品北极星

> 每一名重要武将都在依据自身目标、关系、记忆和当前世界状态行动；每一场战争、结盟、背叛与相遇都能追溯到可验证的世界原因，而不是单纯由脚本或语言模型即兴编造。

## 3. 与《霸王的大陆》的关系

项目继承的是经典三国策略游戏的高层节奏：

- 选君主或选择身份；
- 经营城池；
- 招募与使用武将；
- 外交、战争与统一；
- 清晰的回合/时间推进；
- 复古、紧凑、易理解的战略反馈。

项目不复制任何商业游戏的代码、ROM、地图、头像、音乐、UI、文本、数值表或可识别版式。所有系统、视觉和内容继续原创实现。

## 4. 核心体验

### 4.1 双层世界

**战略层**

- 全国沙盘、城池、道路、地形、势力与军队；
- 时间推进、经济、外交、战争、人口与情报；
- 玩家可快速下达宏观命令。

**在地层**

- 城池、府衙、酒馆、军营、野外营地和战场等局部 3D 场景；
- 玩家可控制角色移动、交谈、调查、拜访和执行任务；
- 场景按模板与世界状态生成，不要求一开始无缝连接全国。

### 4.2 多种身份

长期可支持：

- 君主：经营势力、任命官员、外交与战争；
- 太守：治理一城、处理地方关系与边境压力；
- 武将：效忠、转投、立功、结交与成长；
- 在野人物：游历、受聘、自立或参与地方事件。

第一可玩版本仍以君主模式为主，避免同时开发四套游戏。

### 4.3 活的武将

每名核心 NPC 至少拥有：

- 身份与能力；
- 当前目标；
- 忠诚、野心、恐惧、声望等结构化状态；
- 对人物与势力的关系；
- 已知事实与未知事实；
- 近期事件记忆与长期摘要；
- 可执行行动集合；
- 行动理由与结果记录。

## 5. AI 的正确职责

AI/LLM 负责：

- 从有限候选中规划 NPC 意图；
- 生成符合状态的对话、书信、传闻和事件描述；
- 将多项世界事件组织成有戏剧张力的故事线；
- 为美术和内容生产提供候选，但必须经过资产与版权检查；
- 对低频、重要人物进行更深入推理。

AI/LLM 不负责：

- 直接修改权威世界状态；
- 自行决定战斗数值结果；
- 绕过经济、移动、外交和时间规则；
- 在自由文本中创造不存在的城池、人物、资源或历史事实并直接写入存档；
- 每帧驱动 NPC 或物理模拟。

所有 AI 决策必须通过结构化 Action API：

```ts
type WorldAction =
  | { type: 'TRAVEL'; actorId: string; destinationId: string }
  | { type: 'NEGOTIATE'; actorId: string; targetId: string; proposalId: string }
  | { type: 'RECRUIT'; actorId: string; targetId: string }
  | { type: 'GOVERN'; actorId: string; cityId: string; policyId: string }
  | { type: 'MARCH'; armyId: string; targetCityId: string }
  | { type: 'INVESTIGATE'; actorId: string; subjectId: string };
```

执行流程：

```text
观察世界投影
→ 生成候选意图
→ Schema 校验
→ 规则引擎检查
→ 权威模拟器执行
→ 记录事件
→ AI 生成叙事表达
```

## 6. 建议架构

```text
Authoritative World Simulation
├── WorldState
├── Time / Calendar
├── Economy / Population
├── Factions / Diplomacy
├── Characters / Relationships
├── Armies / Travel / Battles
├── Information / Fog of War
└── Event Log

Agent Runtime
├── Observation Builder
├── Goal Planner
├── Memory Store
├── Action Proposal
├── Action Validator
└── Model Router / Budget

Narrative Layer
├── Dialogue
├── Letters
├── Rumors
├── Chronicle
└── Event Presentation

Three.js Client
├── Strategic Map
├── Local Scene Streaming
├── Character Controller
├── NPC Presentation
└── UI / Interaction
```

### 6.1 权威模拟器

- 必须可在没有云端模型时独立运行；
- 使用 seeded RNG；
- 相同输入和 seed 可重放；
- 存档保存结构化状态和事件日志；
- AI 输出只能成为待验证命令。

### 6.2 Agent Runtime

- 高频普通 NPC 使用规则、utility AI 或小模型；
- 关键武将与关键事件才调用昂贵模型；
- 对话和计划异步预生成并缓存；
- 每个回合/游戏日有调用预算；
- 模型不可用时自动降级，不阻塞游戏。

### 6.3 记忆

记忆分层：

- Working：当前任务与最近交互；
- Episodic：带时间、地点、参与者的事件；
- Semantic：对人物、势力与世界的稳定认知；
- Relationship：好感、信任、恐惧、恩怨和承诺；
- Summary：周期性压缩后的长期经历。

世界事实与角色认知必须分离。NPC 可以不知道真相，也可以被欺骗，但不能让错误认知直接改写世界事实。

## 7. 开发阶段

### Phase 0：完成当前垂直切片

- 美术/UI 定稿；
- 自动化协作与 PR 门稳定；
- 确定性 RNG；
- 核心规则测试；
- 存档迁移和事件日志基础。

### Phase 1：Living World Kernel

- 世界时钟；
- 角色位置与日程；
- 目标、关系、忠诚与野心；
- 外交状态；
- 结构化事件总线；
- 可重放的权威模拟。

验收场景：不调用 LLM，世界连续运行多个游戏月，势力与人物行为合理且可复盘。

### Phase 2：AI Social Sandbox

- 先在文本/面板层实现 AI 交涉；
- 武将记忆、书信、游说、招募、谣言和劝降；
- Action API、Schema 校验与预算器；
- 所有对话必须引用可用世界上下文。

验收场景：两名武将因共同事件建立关系，之后在招募或背叛决策中产生可解释影响。

### Phase 3：单城可探索垂直切片

- 一个原创城市场景模板；
- 玩家角色控制、碰撞、交互；
- 府衙、酒馆、军营三个可进入地点；
- 少量 NPC 的日程和现场对话；
- 战略层与在地层状态同步。

验收场景：玩家从战略地图进入城池，拜访武将并形成一个真实影响战略状态的结果。

### Phase 4：多城与旅途

- 城市模板参数化；
- 区域流式加载；
- 道路、野外事件、营地与侦察；
- 旅行时间与风险；
- 移动端性能预算。

### Phase 5：AI Director 与生成式历史

- 根据世界矛盾选择故事焦点；
- 生成任务链候选；
- 编年史与战报；
- 保证每个事件都有结构化原因、参与者和结果；
- 支持“历史模式”和“架空模式”。

## 8. 明确不做

在 Living World Kernel 验证前，不做：

- 全国无缝第三人称地图；
- 全部城池手工独立建模；
- 每个 NPC 每分钟调用大模型；
- 实时多人；
- 语音驱动全部 NPC；
- AI 直接生成并执行任意代码；
- 无规则约束的无限自由输入；
- 复制原商业游戏的地图、视觉或数值。

## 9. 技术与产品风险

### 范围爆炸

对策：战略层先行；只做一个可探索城市；每个阶段必须有独立可玩验收。

### 世界状态与 AI 对话矛盾

对策：权威状态、观察投影、结构化行动、规则校验、事件日志。

### 成本与延迟

对策：模型分级、缓存、每日预算、批处理、关键角色优先、离线降级。

### 不可复现

对策：seeded RNG、模型输出存档、动作哈希、事件重放、固定测试夹具。

### 内容失控或风格漂移

对策：角色卡、时代词汇约束、输出过滤、事实引用、人工可审查日志。

### 浏览器性能

对策：局部场景加载、对象池、LOD、实例化、Web Worker、固定移动端预算。

## 10. 第一项技术 Spike

在不制作开放世界 3D 场景的前提下，实现一个“AI 武将议事厅”原型：

1. 选择三名现有武将；
2. 从当前 GameState 构建只读 Observation；
3. 每名武将拥有一个目标和三条事件记忆；
4. AI 或可替换规则引擎只能提出 Schema 化行动；
5. Action Validator 接受或拒绝；
6. 权威模拟器执行被接受行动；
7. UI 展示“提议—规则判断—结果—理由”；
8. 模型不可用时使用 deterministic stub；
9. 不修改现有正式玩法路径；
10. 用固定 seed 做自动测试。

该 Spike 只验证核心命题：**AI 能否在不破坏规则一致性的情况下，让现有武将和势力系统产生更有生命力的行为。**

## 11. PR 评审问题

Codex 需要回答：

1. 当前 GameState 到权威 WorldState 的最小迁移路径是什么？
2. 哪些系统必须先纯函数化？
3. 事件日志、seeded RNG、Action API 应怎样落位？
4. 浏览器端、服务端与模型调用边界如何划分？
5. 第一 Spike 的最小文件改动与测试集合是什么？

Antigravity 需要回答：

1. 双层世界如何保持统一视觉语言？
2. 第一个可探索城市场景应采用何种原创视觉抽象？
3. 战略层进入在地层的过渡如何设计？
4. AI NPC 的记忆、关系和意图如何以非侵入方式呈现？
5. 桌面与手机端的交互差异是什么？

## 12. 决策标准

批准本 RFC 不代表立刻开发完整开放世界，只代表：

- 接受长期产品方向；
- 将“活世界内核”设为现有垂直切片之后的核心技术路线；
- 所有 AI 功能遵守“权威模拟器 + 结构化行动 + 可降级”的原则；
- 先验证单城、少数 NPC 和一个闭环，再扩张规模。
